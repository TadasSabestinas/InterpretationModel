About this tool

A coverage report interpretation tool developed as part of a bachelor's thesis on extending JaCoCo's standard metrics with derived measures grounded in the software-testing literature, augmented by an LLM interpretation layer and empirical validation against git bug-fix history.

What it does

  • Single-report analysis — upload a JaCoCo XML and get a full breakdown: composite quality score and grade (A–F), all derived metrics at project, package, class, and method level. Click any package or class to drill in.
  • Hotspot detection — four risk categories (coverage gap risk, branch blind spots, shallow tests, untested complexity) automatically flag the classes most likely to harbour untested behaviour, filtered to classes with cyclomatic complexity ≥ 3.
  • Quality distribution — histogram showing how classes spread across quality score buckets, so you can see whether low scores are isolated outliers or a systemic pattern.
  • Side-by-side comparison — upload two JaCoCo XMLs to compare metrics between projects, or before and after a refactor, with per-metric delta highlighting.
  • Weight sensitivity analysis — each composite weight is perturbed by ±0.10 and Spearman ρ is computed between baseline and perturbed package rankings, answering how much the rankings depend on the chosen weights.
  • PDF report — download a self-contained PDF of the full analysis, suitable for attaching to a review or thesis appendix.
  • AI interpretation — any metric view can be sent to an LLM for a natural-language reading. The exact prompt is always inspectable via the "Show prompt" toggle.

The metrics

Mean(line, branch) — used in quality score

Arithmetic mean of JaCoCo's line and branch coverage percentages. Surfaces the common pathology of high line coverage paired with low branch coverage in a single number. Inozemtseva & Holmes (ICSE 2014) show that coverage types are highly correlated (Kendall τ ≥ 0.91 across five large Java projects), so this metric is included for interpretability rather than independent predictive strength.

Geometric mean of coverage dimensions — used in quality score

Geometric mean of three JaCoCo dimensions: instruction, branch, and line. The geometric mean disproportionately punishes weakness in any single dimension — if any one is very low, the whole score is dragged down. METHOD is intentionally excluded: JaCoCo's method counter includes compiler-generated synthetic and bridge methods (generics, lambdas) that have no source lines, which would penalise well-tested classes for compiler artefacts. A design choice, not drawn from a specific paper.

Weighted line coverage — used in quality score

Line coverage weighted by the number of bytecode instructions per line. A covered line containing a complex expression (many instructions) contributes more than a trivial one-liner. A JaCoCo-compatible proxy for Statement Frequency Coverage. Aghamohammadi et al. (2021) define SFC using runtime execution counts; JaCoCo does not record those. This metric substitutes static instruction density as a coverage-thoroughness proxy — it is an approximation, not SFC itself.

Branch density — display only, not in quality score

Total branches divided by total lines for a class. A structural property, not a coverage metric — it contextualises branch coverage results. High branch coverage in a high-density class is more meaningful than the same percentage in a trivial utility class.

Quality score — the composite

Weighted combination of the three coverage-derived metrics:

0.35 × Mean(L,B) + 0.35 × Geo mean + 0.30 × Wtd line

Rolled up from class to package and project via complexity-weighted averaging: each class's score is weighted by its cyclomatic complexity total. A package with one large complex class scoring 60% and ten trivial classes scoring 100% reports closer to 60%, not 96%. Grades A–F map to 90+, 80+, 70+, 60+, and below.

The AI interpretation layer

AI interpretations are powered by Groq's llama-3.3-70b-versatile via the Groq API. For each request, the relevant metrics are serialised into a structured block and sent with a system prompt instructing the model to be concise, honest about uncertainty, and grounded only in the data shown — it is explicitly told not to invent metrics.

Three prompt types are used: single-target interpretation (project / package / class), side-by-side comparison of two targets, and free-form follow-up questions. The exact prompt sent for any interpretation is always visible via the "Show prompt sent to the model" toggle, which is a deliberate transparency choice for the thesis methodology.

Empirical validation

The tool includes a standalone correlation analysis script (analysis/bugfix_correlation.py) that correlates quality scores against bug-fix commit activity from a project's git history. Applied to Apache Commons Lang (9,329 commits, 338 classes), the composite quality_score achieved Spearman ρ = −0.308 (p < 0.001) against bug-fix commit counts — roughly 83% stronger than raw JaCoCo line coverage (ρ = −0.168), which lost statistical significance entirely on non-trivial classes (CC ≥ 10).

The sensitivity analysis (built into the web UI) found that package rankings under all six ±0.10 weight perturbations maintained Spearman ρ ≥ 0.993, with a maximum rank shift of 1 position — confirming the composite is stable under reasonable weight variation.

Honest limitations

  • The composite score's weights (0.35 / 0.35 / 0.30) are theoretically motivated defaults, not empirically optimised against a labelled dataset.
  • Weighted line coverage uses static instruction density as a proxy for coverage thoroughness. True Statement Frequency Coverage (Aghamohammadi et al., 2021) requires runtime execution counts that JaCoCo does not record.
  • Mean(line, branch) is included for interpretability; Inozemtseva & Holmes (ICSE 2014) show the different coverage types are highly correlated, so combining them adds limited independent predictive power.
  • The bug-fix correlation study uses a JaCoCo snapshot from the current build against the full git history. Classes that were poorly covered historically may have since been improved, which would underestimate the true correlation.
  • AI interpretations are generated by a general-purpose language model and are not validated against ground truth. They should be treated as a starting point for investigation, not as authoritative assessments.
